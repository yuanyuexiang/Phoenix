# 单据防篡改校验规则详解

## 一、校验维度总表

| 维度 | 适用单据 | 校验方法 | 异常含义 |
|------|----------|----------|----------|
| 明细求和 | 报销单/费用清单/发票明细 | 各行同列相加 vs 小计/合计栏 | 明细被增删或合计被改 |
| 汇总公式 | 报销单/结算单 | 声明的等式是否成立 | 某栏金额被改 |
| 隐含值反推 | 含汇总但无明细合计的单据 | 总额反推隐含值，再与明细求和比对 | 明细与汇总不一致 |
| 单位换算 | 差旅/包干/计时单据 | 数量 × 单价/标准 = 行金额 | 天数或标准被改 |
| 日期连续性 | 所有含日期的单据 | 起讫日期合理性、跨年错填 | 日期被改或笔误 |
| 签字完整性 | 审批单/报销单 | 必签栏位是否齐全 | 审批流程缺失 |
| 票号格式 | 增值税发票 | 发票号码位数与校验码规则 | 票号伪造 |

## 二、报销单校验算式模板

以"服务人员报销审批单"为例，标准校验算式：

```
1. 交通费小计 = Σ(各行交通金额)
2. 住宿合计   = Σ(各行住宿金额)
3. 其他合计   = Σ(各行其他金额)
4. 合计发票金额 = 交通费小计 + 住宿合计 + 其他合计   ← 与单据"合计发票金额"栏比对
5. 包干明细合计 = Σ(各行包干总金额)
6. 每行包干总金额 = 该行天数 × 该行包干标准          ← 逐行核对
7. 结算金额 = 包干合计 + 交通费小计                   ← 与单据"结算金额"栏比对
   (或按单据声明的结算公式)
```

异常判定：
- 算式 4 不成立 → 合计发票金额栏被改，或某行明细被改
- 算式 6 某行不成立 → 该行天数或标准被改
- 算式 5 与算式 7 反推的包干合计不一致 → 明细与结算栏存在矛盾（重点疑点）
- 两项汇总公式应同时成立：若 `合计发票金额 = 交通+住宿+其他` 成立，但 `结算金额 = 交通+包干` 不成立，或两者无法同时成立 → 说明三栏中至少一栏被改
- 业务合理性：起点与终点相同的高铁/飞机票（如"长沙-长沙""竹山-竹山"）金额不为 0 时 → 异常，需复核

### 差额解读技巧

差额往往有业务含义，解读后写入结论：
- 差额 = N × 日标准 → 暗示 N 天的包干被核减但明细未同步
- 差额 = 某行金额 → 该行可能被整体删除/新增
- 差额 = 整数倍补贴 → 可能是补贴标准调整未同步

## 三、增值税发票校验算式模板

以"增值税专用发票/普通发票"为例，标准校验算式：

```
1. 金额明细求和 = Σ(各行金额)              ← 与票面"合计金额"栏比对
2. 税额明细求和 = Σ(各行税额)              ← 与票面"合计税额"栏比对
3. 每行金额 = 该行单价 × 数量              ← 逐行核对（允许末位四舍五入）
4. 每行税额 = 该行金额 × 税率              ← 逐行核对（注意四舍五入）
5. 价税合计 = 金额合计 + 税额合计           ← 与票面"价税合计(小写)"比对
6. 大写金额 = 小写金额                      ← 逐字对应
7. 税号校验：购/销双方纳税人识别号位数（国内企业18位）
```

### 税额尾差判定规则（重要）

各行税额若单独四舍五入到分，再求和，可能与票面合计税额差 0.01~0.05 元。
- 差额 ≤ 0.05 元 → 正常四舍五入尾差，标记为 ✅（注明尾差金额）
- 差额 > 0.05 元 → ⚠️ 异常，需人工复核

### 发票字段命名注意（Phoenix 后端）

Phoenix 后端 `invoice` 类型的必填字段名为 `buyer`、`seller`（购买方/销售方名称），
**不是** `buyer_name`/`seller_name`。提交 fields 时务必用 `buyer`/`seller`，否则返回 needs_review。

## 四、doc_no 自动生成规则

### 类型前缀表

| 单据类型 | doc-type | 前缀 |
|----------|----------|------|
| 报销单 | generic/reimbursement | BX |
| 增值税发票 | invoice | FP |
| 合同 | contract | HT |
| 通用/其他 | generic | GEN |

### 生成格式

```
{前缀}-{username}-{YYYYMMDD}-{三位序号}
```

- username：`auth.py --whoami` 返回的 username 字段
- YYYYMMDD：入库当日系统日期
- 三位序号：001 起；若需查当日已有同类编号可先 query 同日前缀的 doc_no 取最大序号 +1

示例：`BX-bob-20260721-001`

## 五、title 自动生成规则

```
{单据类型名}-{主体人/报销人}-{起始日期至结束日期}
```

- 单据类型名：从原件标题提取（如"服务人员报销审批单"）
- 主体人：报销人/申请人/开票方
- 日期范围：单据覆盖的业务日期区间，格式 YYYYMMDD至YYYYMMDD

示例：`服务人员报销审批单-杜永强-20251008至20260101`

## 六、校验结论写入格式

在 content-text 末尾追加：

```
【防篡改/自洽性校验结论】
1. {校验项}：{算式} ✅/⚠️ {比对结果}
2. ...
N. ⚠️ {疑点描述}，差额{金额}，{业务解读}

校验结论：{总体判断}，{建议动作}。
```

同时在 fields 中写 `check_result` 字段，一句话概括。

## 七、实战示例

### 案例 1：服务人员报销审批单（2026-07-21 入库）

明细包干总金额：28350 / 0 / 13950 / 450 / 180
- 明细求和 = 42930
- 天数×标准：63×450=28350✅ 31×450=13950✅ 1×450=450✅ 1×180=180✅ 行内自洽
- 单据结算金额 39145.5 = 交通费 715.5 + 隐含包干 38430
- ⚠️ 42930 ≠ 38430，差额 4500 = 10 × 450（日标准），疑似 10 天包干被核减但明细未改

结论写入 check_result：`明细求和与结算栏差额4500,需复核`

### 案例 2：湖南增值税专用发票（2026-07-21 入库）

6 行货物明细，税率 17%：
- 金额明细：4958.76 + 2531.25 + 278.83 + 1778.15 + 762.66 + 235.78 = 10545.43 ✅ 与合计金额栏一致
- 税额明细：842.96 + 430.31 + 47.40 + 302.29 + 129.65 + 40.08 = 1792.69，票面合计 1792.72，差 0.03 元
  → 0.03 ≤ 0.05，属正常四舍五入尾差 ✅
- 价税合计：10545.43 + 1792.72 = 12338.15 ✅ 与小写一致
- 大写金额：壹万贰仟叁佰叁拾捌圆壹角伍分 = 12338.15 ✅
- 单价×数量：6 行均四舍五入到对应金额 ✅

结论写入 check_result：`金额/税额/价税合计全部自洽,税额明细求和与票面合计差0.03元系四舍五入尾差,未发现改动痕迹`

## 八、特殊情况处理

### 1. 字迹偏小/无法逐行核对

当扫描件分辨率不足、行数多、字迹模糊，导致无法逐行准确读取每一笔明细时：
- 先记录能看清的汇总栏（小计交通、包干、住宿、合计发票金额、结算金额）
- 优先做**汇总栏三项互核**：
  - 合计发票金额 = 交通 + 住宿 + 其他
  - 结算金额 = 交通 + 包干
  - 若两项汇总公式无法同时成立 → 即使明细未逐行核对，也可判定汇总栏存在严重不自洽
- 在 content-text 中写明"因原单字迹偏小，逐行明细无法完全核对；但汇总栏已出现矛盾"
- 总体结论写"需人工核对原始票据"，不建议直接付款

### 2. 同地往返票

报销单中出现起点与终点完全相同的交通记录（如"竹山-竹山""长沙-长沙"），且金额不为 0 时：
- 在业务上属于不合理行程
- 在 content-text 中点名该行，建议作为重点复核项
- 示例：2026-07-21 入库的 page 2 中，第 4 行"竹山-竹山"高铁 18 元、第 7/12 行"长沙-长沙"高铁 1 元，均列入疑点

### 4. 差旅费+包干费合并表的 OCR 误读陷阱（2026-07-21 踩坑）

**典型表格结构**：很多差旅报销单把"差旅费"和"包干费"放在同一张表里，每行**二选一**：

```
| 日期 | 路线 | 交通费-工具 | 交通费-金额 | 出差天数 | 包干标准 | 包干总金额 | 住宿 | 其他 |
| 2026/1/27 | 南京-广德 | 其他 | 0  |    |     |     |   |   |   ← 交通费行
| 2026/1/27 | 桂平-桂平 |   |    | 14 | 450 | 6300 | 1952 |   |   ← 包干行
```

**坑点**：包干行的"交通费-金额"列是**空**的，但容易被误读成"该行包干费=包干标准"或"该行有交通费"。

**反例（2026-07-21 黄智超两单）**：
- 我把"桂平-桂平"行当作交通费 41，实际该行只有包干 6300 → 求和多 41
- 我把"南宁-南宁"行当作交通费 16，实际该行只有包干 500 → 求和多 16
- 两次合计多 57

**正确识别方法**：
1. **逐行判定**：先看每行的"交通费-金额"列是否非空，非空才是交通费行
2. **包干行的金额应填在"包干总金额"列**，"交通费-金额"列留空
3. **求和时只对交通费行求和**，包干行的金额不参与交通费小计
4. 高分辨率识别时，把表格线画出来逐行判读，避免"看到数字就抄"

**校验时增加一步**：
- 若发现某行"交通费-金额"和"包干总金额"**同时**有值，或**同时为空但被填入数字**，几乎可判定为 OCR 误读
- 复核时优先对这种"矛盾行"做高清重读

### 3. 大 PDF 上传失败

- 若单 PDF 过大导致后端 413 / Broken pipe，可拆分为单页图片分别上传
- 每页作为独立单据识别、校验、入库（文件名与原始 PDF 关联）
- 记录于工作日志：原始 PDF 路径、拆页数量、各页 doc-id
