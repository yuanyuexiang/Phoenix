# 单据防篡改与自洽性校验

## 概述

本技能在 Phoenix 文档助手"识别字段 → 入库"之间插入一道**自洽性校验**环节。不只机械抽取字段，而是理解单据的业务逻辑，核对各项数字、日期、签字是否自洽，从而发现单据是否被改动过。校验结论会写入正文 content-text 和 check_result 字段，供后续查询与人工复核。

详细校验规则与示例见 `references/check-rules.md`。

## 触发场景

> **强制规则：发票、报销单、费用清单、结算单等一切涉及费用的票据，入库前必须执行本技能的校验流程，不可跳过。** 费用票据一旦被篡改，直接造成资金损失，校验是防篡改的最后一道关口。

- 经 Phoenix 上传报销单、发票、审批单、费用单等财务/票据类单据后
- 字段提取完成、调用 save 入库之前
- 单据含金额明细、汇总栏、签字栏等可勾稽信息时必做
- 即便单据无明细（如定额发票），也需核对价税合计、大写金额、票号格式等基本自洽性

## 校验工作流

### Step 1: 字段提取（照常进行）

按 Phoenix 标准流程从原件逐项抽取字段值，找不到的留空，不编造。

### Step 2: 金额自洽性校验（核心）

对含明细表的单据，逐项核算：

1. **明细求和校验**：把每行同一列的金额相加，与单据"小计/合计"栏比对。
   - 一致 → 标记 ✅
   - 不一致 → 标记 ⚠️，记录差额，列为疑点
2. **汇总公式校验**：核对单据声明的汇总关系是否成立。
   - 例：合计发票金额 = 交通费 + 住宿 + 其他
   - 例：结算金额 = 包干总金额 + 交通费
   - 公式不成立 → 标记 ⚠️，记录差额
3. **隐含值反推校验**：当汇总栏给出总额但未给明细合计时，反推隐含值，再与明细求和比对。
   - 例：结算金额 39145.5 = 交通费 715.5 + 隐含包干合计 38430；若明细包干求和为 42930，则差额 4500 即为疑点。
4. **单位换算校验**：天数 × 包干标准 = 该行包干总金额，逐行核对。
5. **发票价税校验（增值税发票专用）**：
   - 金额明细求和 = 合计金额栏
   - 税额明细求和 = 合计税额栏（注意：各行税额单独四舍五入后求和，可能与合计栏差 0.01~0.05 元尾差，属正常；超过此范围需警惕）
   - 金额 + 税额 = 价税合计（小写）
   - 大写金额 = 小写金额（逐字对应，注意"圆/元""角""分"写法）
   - 单价 × 数量 = 该行金额（逐行核对，允许末位四舍五入）
   - 票号格式：发票代码 + 发票号码位数是否符合当地规则
   - 购销双方税号位数（国内企业 18 位统一社会信用代码）

发现差额时，尝试解读差额含义（如差额 = N 天 × 日标准，暗示某段天数被核减但明细未改）。

### Step 3: 签字与栏位完整性校验

- 检查必签栏位是否齐全（报销人、审批人、负责人等）
- 检查日期连续性（起讫日期是否合理、有无跨年错填）
- 检查附件张数、外协单位等辅助字段是否填写

### Step 4: 自动生成 doc_no 与 title

当单据原件未填写编号(doc_no)或标题(title)，而后端要求必填时，按以下规则自动生成：

- **doc_no**：`{类型前缀}-{登录用户}-{YYYYMMDD}-{当日序号}`
  - 类型前缀：报销 BX / 发票 FP / 合同 HT / 通用 GEN
  - 登录用户：取 `auth.py --whoami` 返回的 username
  - YYYYMMDD：入库日期（取系统当前日期）
  - 当日序号：001 起，若当日已有同类单据则递增
  - 示例：`BX-bob-20260721-001`
- **title**：`{单据类型}-{报销人/主体}-{起始日期至结束日期}`
  - 示例：`服务人员报销审批单-杜永强-20251008至20260101`

生成规则目的：保证 doc_no/title 唯一且可按用户、时间检索。

### Step 5: 写入校验结论

在 content-text 末尾追加【防篡改/自洽性校验结论】段落，逐条列出：
- 每项校验的算式与 ✅/⚠️ 标记
- 发现的差额及其可能的业务解读
- 总体结论（自洽 / 存在疑点需人工复核）

同时在 fields 里加 `check_result` 字段，用一句话概括（如"明细求和与结算栏差额4500,需复核"）。

### Step 6: 入库

带完整 fields（含 doc_no、title、check_result）和含校验结论的 content-text 调用 save。
- 若仅 doc_no/title 缺失（无金额疑点）→ 补齐后直接 save，通常可 saved
- 若存在金额疑点 → 仍正常 save（不擅自 --force），让 status 反映校验状态；校验结论已写入正文供人工复核

## 校验原则

- **只核对、不改写**：保持单据原始数字写法，不替用户"修正"金额。校验是发现疑点，不是改账。
- **差额要量化**：不只说"不一致"，要算出具体差额并尝试解读含义。
- **结论要可追溯**：校验算式写进 content-text，任何人查文档都能复核。
- **不阻断入库**：除非用户明确要求拦截，否则校验疑点不阻止入库，只标记供复核。
